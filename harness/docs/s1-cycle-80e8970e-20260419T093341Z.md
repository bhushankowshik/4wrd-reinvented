## 1. Crystallised problem

## Problem name
AI-Assisted Solutioning Estimator

## Crystallised statement
Solo founders operating within a 4WRD delivery cycle lack a fast, repeatable way to estimate the effort required for a solutioning engagement from free-text problem input alone. There is no tool that infers likely failure modes from that input and returns dual-mode estimates — conventional and 4WRD — expressed in person-days, without dependency on historical data. This tool must produce its output as structured markdown, consumable immediately in Notion (primary) and Google Doc (secondary), so that the estimate artefact is ready for use without reformatting. The estimator must work for a solo founder with no team and no prior engagement history to draw on.

## In scope
- Free-text input only
- AI failure mode inference
- Dual-mode person-day estimates
- Structured markdown output
- Notion primary
- Google Doc secondary
- No historical data
- Solo founder context

## Out of scope
- Project management integrations
- Team workflows
- Historical data dependency
- Non-markdown output formats
- Post-estimate tracking

## Success criteria
1. Zero-question output — the tool produces an estimate from free-text input alone, without asking the user any clarifying questions
2. Failure mode inference — the output includes AI-inferred failure modes derived solely from the free-text input
3. Dual-mode estimates in person-days — the output contains both a conventional delivery estimate and a 4WRD delivery estimate, each expressed in person-days
4. Notion rendering — the structured markdown output renders correctly and is immediately usable in Notion without reformatting
5. Google Doc rendering — the structured markdown output renders correctly and is immediately usable in Google Doc without reformatting
6. No historical data required — the tool operates without access to any prior engagement history or team data

## Primary stakeholders
- Solo founders operating within a 4WRD delivery cycle

## Key uncertainties
- **KU1** Person-days range vs point — whether estimates are expressed as a point value or a range has not been resolved
- **KU2** Failure mode count — the expected or maximum number of inferred failure modes per estimate has not been specified
- **KU3** 4WRD methodology — the precise definition and parameters of the 4WRD delivery model used to generate the 4WRD-mode estimate have not been confirmed
- **KU4** Markdown dialect precedence — which markdown dialect (e.g. Notion-flavoured vs standard CommonMark) takes precedence when the two targets require different formatting treatment has not been decided
- **KU5** Filename convention — the naming convention for the output artefact file has not been specified
