## 1. Crystallised problem

## Problem name
AI-Assisted Solutioning Estimator: Failure-Mode-Aware, Dual-Mode Effort Estimation for Client-Ready Software Delivery Scoping (v1)

---

## Crystallised statement

Software delivery estimation fails predictably because six structural failure modes — scope underestimation, missing non-functional requirements, compliance overhead omission, rework underestimation, optimistic availability assumptions, and implicit unquantified risk — are routinely invisible in conventional estimation workflows until overrun has already occurred. An AI-Assisted Solutioning Estimator accepts free-text problem and delivery constraint input, infers failure mode risk and complexity signals through AI analysis without clarifying questions, and produces a structured estimate containing role breakdown, phase durations, risk-adjusted effort ranges (expressed in person-days), and an AI-generated assumption register presented for human review before the output is finalised. The tool renders a side-by-side dual-mode comparison — conventional delivery versus 4WRD-governed delivery, where the 4WRD model applies a working assumption of approximately 40% human effort reduction per phase through AI agent contribution at producing and reviewing roles, offset by 10–15% governance overhead — delivered as structured markdown suitable for direct copy-paste into a client document or presentation.

---

## In scope

- **Input handling:** Free-text problem description and delivery constraints accepted as sole input; no clarifying questions issued before first output is produced.
- **Failure mode inference:** AI analysis of free-text input to infer presence and severity of each of the six confirmed failure modes: scope underestimation, missing NFRs, compliance overhead omission, rework underestimation, optimistic availability, and implicit unquantified risk.
- **Effort estimation unit:** Person-days exclusively. No cost conversion, no rate application, no currency figures.
- **Risk-adjusted ranges:** AI-inferred heuristic multipliers derived from scope complexity signals detected in the input; applied to produce low/mid/high effort range bands per phase and role.
- **Assumption register:** AI-generated list of assumptions surfaced explicitly and presented to the human for review and acceptance/amendment before the estimate is treated as final.
- **Dual-mode comparison:** Side-by-side estimate output for (a) conventional delivery and (b) 4WRD-governed delivery.
- **4WRD effort model (v1 working assumption):** ~40% reduction in human effort per phase attributable to AI agents operating in producing and reviewing roles; 10–15% governance overhead added back. Both figures treated as heuristic starting values subject to dogfood validation.
- **Output format:** Structured markdown with clearly delineated sections, suitable for copy-paste into client documents or presentation decks without reformatting.
- **Primary user:** Solo founder or delivery lead operating without a dedicated estimation team.

---

## Out of scope

- Cost conversion, billing rate application, or any monetary output — deferred post-v1.
- Clarifying question loops or multi-turn input refinement before first output generation.
- Integration with project management tooling, ticketing systems, or time-tracking platforms.
- Custom role taxonomy configuration by the user in v1.
- Validation or calibration of the 4WRD effort model against historical delivery data — this is a dogfood exercise, not a v1 feature.
- Compliance framework-specific overhead lookup tables (compliance risk is inferred generically from input signals, not mapped to named regulatory regimes in v1).
- Presentation rendering beyond markdown (e.g., PDF export, slide deck generation, interactive visualisation).

---

## Success criteria

1. **Zero-question first output:** Given any free-text input describing a software delivery problem, the tool produces a complete structured estimate without issuing a single clarifying question prior to first output.
2. **All six failure modes addressed:** Every output explicitly surfaces a risk signal or confirmed-low assessment for each of the six failure modes, traceable to signals in the input text.
3. **Risk-adjusted range produced:** Every estimate contains a low/mid/high person-day range per phase, with the multiplier logic visible or inferable from the assumption register.
4. **Assumption register present and reviewable:** The output includes a discrete assumption register section listing all AI-generated assumptions; a human can read, accept, or amend each item before treating the estimate as final.
5. **Dual-mode comparison rendered:** Output contains a side-by-side or clearly separated conventional vs. 4WRD estimate, with the 4WRD figures reflecting the ~40% reduction and 10–15% governance overhead model.
6. **Markdown fidelity:** Output renders correctly in standard markdown viewers and can be pasted into a Google Doc, Notion page, or PowerPoint notes field without structural degradation.
7. **Person-days only:** No monetary figures appear anywhere in the output.
8. **Dogfood baseline captured:** The 4WRD effort model assumptions (40% reduction, 10–15% overhead) are labelled as working assumptions in the output, creating a verifiable baseline for future calibration.

## Key uncertainties

> These are open items the human must resolve before or during build. They do not block the problem statement but will constrain implementation decisions.

1. **Assumption register review mechanism:** Is the human review of AI-generated assumptions a synchronous in-tool interaction (e.g., the tool pauses, presents assumptions, waits for confirmation before rendering the final estimate), or is the assumption register simply included in the output for post-hoc review? The current statement implies the latter but does not foreclose the former. This affects UX architecture.

2. **Heuristic multiplier transparency:** Should the risk-adjusted multipliers be shown explicitly in the output (e.g., "scope complexity signal detected → ×1.4 applied to discovery phase"), or is the range the only visible output with the multiplier logic internal? Client-facing documents may require one or the other.

3. **4WRD model floor/ceiling:** The 40% reduction and 10–15% overhead are working assumptions. Is there a defined floor (e.g., the tool will never estimate a 4WRD phase at less than X% of conventional) to prevent implausible outputs before dogfood data is available?

4. **Failure mode inference confidence signal:** If the AI analysis cannot confidently infer a failure mode risk level from the input (e.g., input is too sparse for compliance overhead assessment), does the output flag this as "insufficient signal" or default to a conservative non-zero risk? The current statement does not specify.

5. **Assumption register ownership post-output:** Once the human amends the assumption register, does the tool regenerate the estimate to reflect amended assumptions, or is amendment a manual annotation? This is a v1 scope boundary that needs explicit closure.

6. **Markdown target environment:** "Suitable for copy-paste into a client document or presentation" covers a wide range of renderers. If a specific primary target (e.g., Notion, Google Slides, Microsoft Word) is nominated, heading hierarchy and table syntax choices may need to be constrained.
