## 1. Crystallised problem

## Problem name
AI-Assisted Solutioning Estimator: Failure-Mode-Aware, Dual-Mode Effort Estimation for Client-Ready Software Delivery Scoping

---

## Crystallised statement

Software delivery estimation fails predictably because six structural failure modes — scope underestimation, missing non-functional requirements, compliance overhead omission, rework underestimation, optimistic availability assumptions, and implicit unquantified risk — are routinely invisible in conventional estimation workflows until overrun has already occurred. An AI-Assisted Solutioning Estimator accepts a free-text problem description and delivery constraints as input, explicitly interrogates each failure mode, and produces a structured estimate containing role breakdown, phase durations, risk-adjusted effort ranges, and a declared assumption register. The tool operates in dual mode — conventional delivery estimation alongside 4WRD-governed delivery estimation — rendered as a side-by-side comparison that accounts for AI agents as first-class role contributors, reduced human effort per phase, and added governance overhead. The primary user is a solo founder or delivery lead who must produce a defensible, client-conversation-ready estimate without access to a full estimation team or historical project data.

---

## In scope

- **Input handling:** Accept free-text problem description and delivery constraints as the primary input mechanism; no structured intake forms required as prerequisite
- **Failure mode surfacing:** Explicit interrogation and flagging of all six known failure modes against the input scope before any estimate is produced
- **Estimate output structure:**
  - Role breakdown (human and AI agent roles, differentiated by mode)
  - Phase durations with named phases
  - Risk-adjusted effort ranges (optimistic / realistic / pessimistic)
  - Assumption register (explicit, numbered, client-readable)
- **Dual-mode estimation:**
  - Conventional delivery mode: human-staffed roles, standard delivery phases, no AI agent contribution
  - 4WRD-governed delivery mode: AI agents as first-class contributors, reduced human effort per phase, governance overhead added as explicit cost
  - Side-by-side comparison of both modes in a single output artefact
- **Client-ready formatting:** Output structured for direct use in client conversations without post-processing
- **Solo operator use case:** Tool must be operable by a single founder or delivery lead without a supporting estimation team

---

## Out of scope

- Project management or task tracking after estimate is accepted (this tool stops at estimate production)
- Integration with external time-tracking, billing, or resource management systems
- Real-time team availability calendars or HR system feeds
- Automated scope discovery from third-party artefacts (e.g., existing PRDs, Jira backlogs) — input is human-authored free text
- Pricing or commercial rate card generation (effort only, not cost conversion)
- Regulatory compliance determination (compliance overhead is flagged as a risk surface, not resolved by the tool)
- Multi-tenant team collaboration features in this iteration
- Historical data learning or feedback loops from completed project actuals (this iteration)

---

## Success criteria

1. **Failure mode coverage:** Every estimate output contains a named, non-empty entry against each of the six failure modes — absence of a failure mode must be an explicit declared assumption, not a silent omission
2. **Dual-mode output completeness:** Every estimate produces both a conventional and a 4WRD-governed variant; neither mode can be output without the other; side-by-side comparison is always present
3. **Assumption register present:** Every estimate output contains a numbered assumption register with at least one entry; zero-assumption outputs are treated as a system defect
4. **Role differentiation in 4WRD mode:** 4WRD output explicitly labels which roles are AI agents, which are human, and quantifies the governance overhead cost in effort units
5. **Risk-adjusted ranges:** All effort figures are expressed as ranges (not point estimates); ranges are labelled with their basis (optimistic / realistic / pessimistic or equivalent)
6. **Client-conversation readiness:** A solo founder or delivery lead can present the output directly to a client without reformatting; verified by usability test with at least one solo-founder persona
7. **Input latency tolerance:** The tool produces a structured estimate from a free-text input of 50–500 words without requiring the user to answer clarifying questions before first output is shown
8. **Scope underestimation detection rate:** In a sample of 10 test scopes with known hidden scope gaps, the tool flags a scope underestimation risk in at least 8 of 10 cases

## Key uncertainties

The human must resolve the following before this problem statement is considered stable:

1. **4WRD effort model quantification:** The 4WRD mode requires AI agent effort contributions and governance overhead to be expressed in effort units. Is there an existing effort model or reference calibration for how much human effort AI agents displace per phase, and what governance overhead costs in hours? Without this, the 4WRD side of the comparison cannot be computed — it can only be structured.

2. **Failure mode interrogation mechanism:** Does the tool *ask the user targeted questions* to surface failure modes, or does it *infer failure mode risk from the free-text input using AI analysis*, or both? This determines the UX architecture entirely.

3. **Risk-adjusted range methodology:** What is the source of the range calibration? Options include: heuristic multipliers (e.g., +30% / -10%), statistical models, historical analogue data, or expert-rule systems. The tool cannot produce credible ranges without a declared methodology.

4. **Assumption register authorship:** Are assumptions AI-generated from the input and presented for human review, or are they human-entered with AI prompting? The trust and liability profile of the output differs materially depending on the answer.

5. **NFR and compliance detection scope:** The tool flags missing NFRs and compliance overhead as failure modes. Does it have a predefined taxonomy of NFRs and compliance domains to check against (e.g., GDPR, SOC2, accessibility), or is detection purely inference-based from context?

6. **Output format and delivery channel:** Is the client-ready output a PDF export, a shareable web view, a structured document, or something else? Format constraints affect what "client-conversation ready" means in practice.

7. **Contradiction flag — scope vs. solo operator:** The direction specifies "delivery teams" as users in one place and "solo founder or delivery lead" as the primary user in another. If the primary user is solo, is team-based collaborative use a current or future requirement? This affects whether multi-user session handling is deferred or excluded.
